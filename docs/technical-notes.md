# 대나무숲 기술 노트

이 문서는 기능을 구현하면서 선택한 설계 의도와 트러블슈팅 포인트를 누적하기 위한 기록이다. 단순히 사용 기술을 나열하기보다, 왜 그런 구조를 택했는지와 이후 배포/확장 시 어떤 점을 고려했는지를 정리한다.

## 프로젝트 방향

대나무숲은 로그인 없는 익명 게시판이다. 핵심 요구사항은 글/댓글 작성자가 익명으로 보이되, 같은 브라우저에서는 내가 쓴 글과 댓글만 수정/삭제할 수 있어야 한다는 점이다. 그래서 사용자 계정 모델을 먼저 도입하지 않고, 쿠키 기반 익명 세션을 소유권 판단의 최소 단위로 사용했다.

이 방향은 초기 구현 비용을 낮추면서도 게시판의 핵심 경험을 유지한다. 나중에 회원 기능을 붙이더라도 `ownerSessionId` 중심의 소유권 판단을 사용자 식별자로 교체하거나 병행할 수 있어, 현재 구조가 완전히 버려지는 형태가 아니다.

## 백엔드 설계 의도

### Controller, Service, Repository 책임 분리

Spring 계층은 HTTP 요청 처리, 비즈니스 규칙, 데이터 접근을 분리하는 쪽으로 구성했다.

- Controller는 요청/응답 매핑과 세션 식별자 확보에 집중한다.
- Service는 글/댓글 작성, 수정, 삭제, 좋아요, 이미지 검증 같은 도메인 규칙을 담당한다.
- Repository는 JPA 기반 데이터 조회와 집계 쿼리만 담당한다.
- DTO는 API 입출력 형태를 고정해 Entity가 프론트엔드 계약에 직접 노출되지 않게 한다.

이렇게 나눈 이유는 기능이 늘어날수록 Controller에 조건문과 검증이 몰리는 문제를 피하기 위해서다. 예를 들어 게시글 수정 시 소유권 검증, 닉네임 기본값 처리, 이미지 개수 제한은 HTTP와 무관한 규칙이므로 `PostService`에 둔다. 반대로 파일 업로드의 multipart 처리처럼 HTTP 형태와 가까운 부분은 Controller에서 진입점을 만들고, 저장 정책은 별도 서비스로 넘긴다.

### 쿠키 기반 익명 세션

`AnonymousSessionService`는 `anonymous_session_id` 쿠키를 발급하고 읽는다. 쿠키는 `HttpOnly`, `SameSite=Lax`, `path=/`, 30일 만료 정책을 가진다.

이 선택의 의도는 로그인 없는 UX를 유지하면서도 서버가 작성자 소유권을 판단할 수 있게 하는 것이다. 프론트엔드 localStorage만으로 소유권을 판단하면 사용자가 값을 조작할 수 있고, 서버 API를 직접 호출했을 때 보호가 깨진다. 그래서 최종 수정/삭제 권한 검증은 서버에서 세션 ID와 작성 시 저장된 owner 값을 비교한다.

배포 시에는 HTTPS 환경에서 `app.cookie.secure=true`로 전환할 수 있게 설정값으로 분리했다. 로컬 H2 환경과 배포 환경의 차이를 코드 변경이 아니라 설정 변경으로 다루기 위한 선택이다.

### 좋아요 집계와 likedByMe 계산

게시글 목록 응답에는 좋아요 수, 내가 좋아요를 눌렀는지, 댓글 좋아요 수까지 같이 내려간다. 단순하게 각 게시글/댓글마다 count 쿼리를 반복하면 데이터가 늘어날 때 N+1 문제가 생기기 쉽다.

그래서 게시글 ID와 댓글 ID 목록을 먼저 모은 뒤, repository 집계 쿼리로 좋아요 수와 내가 누른 대상 목록을 한 번에 구해 DTO 조립 시 합친다. 이 방식은 초기 규모에서도 코드가 조금 길어지는 대신, 게시판 목록처럼 자주 호출되는 API의 확장성을 확보한다.

### 이미지 첨부 모델

이미지는 `PostImage`로 게시글과 분리했다. 게시글 하나에 최대 10장을 둘 수 있고, 각 이미지는 `sourceType`, `url`, `originalFilename`, `imageOrder`를 가진다.

분리한 이유는 다음과 같다.

- 게시글 본문과 이미지 목록의 생명주기를 분리할 수 있다.
- URL 이미지와 직접 업로드 이미지를 같은 API 모델로 다룰 수 있다.
- 이미지 순서, 원본 파일명, 저장 방식 같은 메타데이터를 나중에 확장하기 쉽다.
- PostgreSQL로 전환해도 게시글 테이블에 이미지 컬럼을 여러 개 추가하는 구조보다 자연스럽다.

`sourceType`을 둔 것도 배포를 고려한 선택이다. 지금은 로컬 파일 저장과 외부 이미지 URL을 함께 지원하지만, 나중에 S3를 붙이면 업로드 저장소 구현과 공개 URL 생성 정책만 바꾸고 게시글 API 모델은 크게 흔들지 않을 수 있다.

### 로컬 파일 저장과 추후 S3 전환 고려

`PostImageStorageService`는 현재 multipart 이미지를 로컬 디렉터리에 저장하고 `/uploads/post-images/...` 형태의 공개 경로를 반환한다. 저장 디렉터리, 공개 경로, 최대 파일 크기는 설정값으로 분리했다.

초기에는 H2/local 환경에서도 동작해야 하므로 로컬 파일 저장이 가장 단순하다. 다만 Controller나 PostService가 파일 시스템 세부 구현을 직접 알지 않게 `PostImageStorageService`로 분리해 두었다. 추후 S3를 도입할 때는 이 서비스를 인터페이스화하거나 구현체를 교체해 저장소 변경 범위를 줄일 수 있다.

트러블슈팅 포인트는 엑박 문제였다. 업로드 자체는 성공했지만 브라우저가 `/uploads/...` 경로를 정적 리소스로 받을 수 없으면 이미지가 깨진다. 그래서 저장 경로와 공개 URL 경로를 구분하고, 로컬/배포 환경에서 정적 파일 서빙 경로를 맞출 수 있게 구성했다. nginx를 붙이면 `/uploads/`를 애플리케이션 또는 파일 저장 경로로 프록시/서빙하는 식으로 확장할 수 있다.

### 서버 검증을 프론트 검증보다 우선

프론트엔드에서도 최대 10장 제한과 파일 선택 제한을 제공하지만, 최종 제한은 서버에서 다시 검증한다. URL 이미지는 http/https만 허용하고, 업로드 이미지는 서버가 발급한 `/uploads/post-images/` 경로만 허용한다.

프론트 검증은 UX를 위한 것이고, 서버 검증은 데이터 무결성과 API 보호를 위한 것이다. 이 둘을 분리해 둬야 브라우저 외부에서 API를 호출하거나 프론트 버그가 생겨도 핵심 규칙이 유지된다.

## 프론트엔드 설계 의도

### App.tsx 비대화 방지

초기에는 게시판 상태와 이벤트가 `App.tsx`에 몰리기 쉽다. 이를 줄이기 위해 API 호출은 `boardApi`, 상태 전이는 `boardReducer`, 날짜/textarea/인기글 판단 같은 UI 유틸은 `boardUi`, 화면 조각은 components로 나눴다.

이 구조의 장점은 변경 범위가 좁아지는 것이다. 예를 들어 페이지네이션 UI를 바꿔도 reducer의 페이지 상태 전이와 API 계약은 건드리지 않는다. 반대로 이미지 첨부 API가 바뀌면 `boardApi`와 이미지 관련 컴포넌트 중심으로 보면 된다.

### Reducer 기반 게시판 상태 관리

게시글 목록, 상세 열림 상태, 작성 중인 글, 수정 중인 글, 댓글 draft, 삭제 확인 상태, 페이지 상태가 한 화면에 같이 존재한다. 이를 여러 `useState`로 흩어두면 액션 간 순서와 초기화 조건을 추적하기 어려워진다.

그래서 `boardReducer`에 상태 전이를 모았다. 예를 들어 글 등록 성공 후에는 내용, 이미지, 본문 표시 옵션을 같이 초기화해야 한다. 상세 글이 삭제되거나 목록 새로고침 후 사라졌을 때 상세 상태도 닫아야 한다. 이런 상태 전이를 reducer action으로 표현하면 UI 컴포넌트는 화면 이벤트를 dispatch하는 역할에 집중할 수 있다.

### API 레이어 분리

`boardApi`는 axios 기반 HTTP 호출을 한 곳에 모은다. 컴포넌트가 URL 문자열과 HTTP 메서드를 직접 알지 않게 하려는 의도다.

이렇게 두면 API prefix, 인증 방식, 에러 처리 정책, 응답 타입이 바뀌어도 수정 지점이 분산되지 않는다. 특히 이미지 업로드는 JSON 요청이 아니라 `FormData` multipart 요청이므로, 일반 게시글 생성/수정과 분리된 API 함수로 두는 것이 유지보수에 유리하다.

### 이미지 입력 UX 변경

처음에는 이미지 URL 입력칸과 파일 선택을 모두 제공했다. 하지만 모바일에서는 URL을 직접 입력하는 흐름이 어색하고, 데스크톱에서도 사용자는 이미지를 복사/붙여넣기하거나 드래그하는 경우가 많다.

그래서 작성 textarea에서 다음 입력을 지원하도록 바꿨다.

- 이미지 파일 붙여넣기
- 이미지 파일 드래그 앤 드롭
- 직접 이미지 URL 붙여넣기 또는 드롭
- 일반 링크는 본문 텍스트로 유지

여기서 모든 URL을 이미지로 바꾸지 않은 이유가 중요하다. 일반 링크까지 이미지 첨부로 처리하면 사용자가 링크를 글에 남기고 싶은 경우를 방해한다. 그래서 확장자가 이미지로 판단되는 직접 이미지 URL만 첨부로 처리하고, 나머지는 본문 텍스트 흐름을 유지한다.

### 목록/상세 이미지 표시

게시글 목록에서는 이미지가 있으면 작은 미리보기를 보여주고, 3장 이상이면 일부만 보여준 뒤 추가 개수를 표시하는 방식으로 설계했다. 상세 화면에서는 본문 이미지 표시 옵션에 따라 크게 보여준다.

목록과 상세의 목적이 다르기 때문이다. 목록은 빠르게 훑고 들어갈 글을 고르는 화면이라 이미지가 너무 커지면 글 흐름을 방해한다. 상세는 게시글 하나에 집중하는 화면이므로 이미지 표현을 더 크게 가져갈 수 있다.

### 작성창과 목록 카드 레이아웃 정리

작성창은 모바일에서 반복적으로 쓰는 영역이므로 세로 공간을 줄이는 방향으로 조정했다. 사진 첨부 설명을 입력칸 주변에 길게 노출하지 않고, `+` 버튼을 눌렀을 때 필요한 첨부 메뉴만 열리게 했다. 첨부된 사진 미리보기는 입력줄 아래가 아니라 위에 배치해, 사용자가 글을 작성하기 전에 현재 첨부 상태를 먼저 확인할 수 있게 했다.

목록 카드는 `인기 여부/닉네임`, `작성 시간`, `사진+본문`, `좋아요/댓글/사진 수`의 순서로 정리했다. 사진 영역은 고정 컬럼이 아니라 실제 미리보기 폭만 차지하게 해, 이미지가 1장뿐일 때 본문과 불필요하게 멀어지는 문제를 줄였다.

### 댓글 접기와 알림 상태 관리

댓글은 상세 화면에서 처음부터 전체 목록을 펼치지 않고 댓글 수 버튼만 보여준 뒤, 사용자가 누르면 목록과 입력폼을 함께 펼치도록 했다. 게시글 본문을 먼저 읽는 흐름을 방해하지 않고, 댓글을 확인하거나 작성하려는 순간에만 추가 UI가 열리게 하려는 의도다.

댓글 알림은 서버 푸시가 아니라 현재 브라우저의 `localStorage`에 읽은 댓글 ID를 저장하는 방식으로 시작했다. 첫 로딩 시 이미 존재하던 댓글은 읽은 것으로 기준선을 잡고, 이후 내가 작성한 글에 새로 붙은 댓글만 상단 알림 후보로 계산한다. 이 방식은 별도 알림 테이블이나 Redis 없이도 로컬 익명 게시판 단계에서 필요한 피드백을 제공한다.

나중에 여러 기기 동기화, 알림 만료, 읽음 상태 공유가 필요해지면 서버 저장 방식이나 Redis TTL 기반 구조로 옮길 수 있다. 현재 구조는 초기 UX 검증용으로 가볍게 시작한 선택이다.

### 브라우저 기본 동작 존중

삭제 확인은 커스텀 토스트가 아니라 기본 alert/confirm 성격의 흐름으로 단순화했다. 모바일 뒤로가기와 브라우저 back도 고려해 상세 화면을 hash 기반으로 열도록 했다.

이 선택은 화려한 UI보다 예측 가능한 동작을 우선한 것이다. 특히 모바일 브라우저에서는 사용자가 back 버튼을 눌렀을 때 앱 내부 상태만 바뀌는지, 브라우저 히스토리와 같이 움직이는지가 체감 품질에 크게 영향을 준다.

## 배포 전 고려사항

### PostgreSQL 전환

현재 H2/local 환경에서 동작하지만, 게시글/댓글/좋아요/이미지 Entity를 분리해 둔 상태라 PostgreSQL 전환 시 테이블 모델은 비교적 자연스럽게 옮길 수 있다. 확인해야 할 부분은 시간 타입, ID 생성 전략, JPA ddl-auto 정책, 운영 마이그레이션 방식이다.

운영에서는 `ddl-auto=create/update`에 의존하기보다 Flyway 같은 마이그레이션 도구를 도입하는 편이 안전하다.

### Redis 도입 후보

현재 익명 세션은 쿠키 값 자체를 식별자로 사용하므로 Redis가 필수는 아니다. 다만 다음 요구사항이 생기면 Redis 도입을 검토할 수 있다.

- 세션 강제 만료/차단
- 인기글 캐싱
- 좋아요/조회수 같은 높은 빈도 카운터 캐싱
- 알림 읽음 상태의 TTL 관리

즉 Redis는 지금 당장 넣는 기술이 아니라, 병목이나 TTL 기반 상태 관리가 실제로 필요해질 때 도입하는 것이 유지비 측면에서 낫다.

### nginx와 업로드 파일

로컬에서는 Spring이 업로드 파일을 저장하고 공개 경로로 접근하게 만들 수 있다. 배포에서는 nginx가 정적 업로드 파일을 직접 서빙하거나, `/uploads/` 경로를 Spring으로 프록시하는 선택지가 있다.

장기적으로는 S3 같은 오브젝트 스토리지를 붙이는 편이 서버 인스턴스 교체와 스케일 아웃에 유리하다. 현재 이미지 모델과 저장 서비스 분리는 이 전환을 염두에 둔 구조다.

## 면접에서 말하기 좋은 포인트

- 로그인 없는 익명 서비스에서도 서버 기준 소유권 검증을 유지하기 위해 쿠키 기반 익명 세션을 설계했다.
- Controller에 비즈니스 규칙을 두지 않고 Service에서 검증과 소유권 판단을 처리해 API 진입점이 늘어나도 규칙이 중복되지 않게 했다.
- 좋아요 수와 likedByMe를 목록 응답에 포함하면서 N+1 count 조회를 피하기 위해 ID 목록 기반 집계 조회로 조립했다.
- 이미지 첨부는 URL과 업로드를 같은 `PostImage` 모델로 추상화해 S3 같은 저장소 전환 가능성을 남겼다.
- 프론트엔드는 reducer/API/component/util로 나눠 `App.tsx`가 화면 전체의 모든 세부 구현을 떠안지 않게 했다.
- 모바일 UX를 고려해 URL 입력칸보다 붙여넣기/드래그/파일선택 중심의 이미지 첨부 흐름으로 바꿨다.
- 목록 카드와 작성창을 사용 흐름 기준으로 재배치해, 스캔과 반복 작성에 드는 화면 비용을 줄였다.
- 댓글 알림은 초기에는 브라우저 로컬 읽음 상태로 구현하고, 다중 기기/TTL 요구가 생기면 서버나 Redis로 옮길 수 있게 분리했다.
- 프론트 검증은 사용자 경험용, 서버 검증은 데이터 무결성용으로 나눠 API 직접 호출에도 핵심 규칙이 유지되게 했다.

다음 기능을 구현할 때 이 문서에 함께 남길 만한 항목이다.

## AI 글쓰기 초안 생성

- 작성창의 `+` 메뉴에서 `AI 글쓰기`를 선택하면 기존 본문을 보존한 채 프롬프트 입력 모드로 전환한다.
- 프론트엔드는 `POST /api/ai/drafts`만 호출하며, 생성 결과를 게시하지 않고 기존 게시글 본문 상태에 넣어 사용자가 검토·수정하게 한다.
- 취소 시 진행 중인 브라우저 요청을 `AbortController`로 중단하고 AI 모드를 닫는다. 이미 작성하던 본문은 덮어쓰지 않는다.
- 백엔드는 `AI_PROVIDER` 설정으로 Gemini와 OpenAI를 선택하며 기본값은 `gemini`다. 프론트엔드 API 계약은 provider와 무관하게 `POST /api/ai/drafts`로 유지한다.
- Gemini는 `GEMINI_API_KEY`와 공식 `generateContent` REST API를 사용한다. OpenAI 구현도 `OPENAI_API_KEY`와 Responses API 방식으로 남겨 환경변수만 바꿔 복귀할 수 있다.
- 입력은 2000자로 제한하고 모델에는 1800자 이내 게시글 본문을 지시한다. Gemini의 생각 토큰과 본문 토큰이 같은 출력 예산을 소모할 수 있어 기본 예산을 8192 tokens로 두고, 추론 수준은 `GEMINI_THINKING_LEVEL`의 기본값 `medium`을 사용한다.
- 기본 instruction은 사용자가 칼럼·에세이·후기·정보글·공지·요약 같은 형식을 지정하면 해당 장르를 최우선으로 따른다. 형식과 분량이 모두 미지정된 일반 커뮤니티 글에만 300~700자, 4~8문장, 2~4문단 기본값을 적용한다.
- 실제 키 주입 및 provider 전환 절차는 `docs/ai-provider-setup.md`에 기록한다. API 키는 브라우저 번들·저장소·요청 payload에 노출하지 않는다.
- AI 결과처럼 React 상태에서 본문이 프로그램식으로 바뀌는 경우에도 `useLayoutEffect`에서 textarea 높이를 다시 계산한다. 생성 요청의 지연은 키나 프롬프트를 기록하지 않고 provider와 소요 시간만 서버 로그로 확인한다.
- 고정 작성창은 `ResizeObserver`로 실제 높이를 측정해 `--composer-reserved-height`를 갱신한다. 본문이 길어져도 목록 마지막 항목과 페이지네이션이 작성창 아래에 가려지지 않으며, AI 생성 중에는 프롬프트 영역에 회전 상태와 경과 초를 표시하고 취소는 유지한다.
- Gemini `finishReason`이 `MAX_TOKENS` 또는 다른 비정상 종료이면 일부 텍스트가 있더라도 성공으로 반환하지 않는다. 서버 로그에는 `finishReason`, 생각/본문/전체 토큰 수를 남겨 모델 품질 문제와 요청 한도 문제를 구분한다.

## 전용 게시글 작성 화면

- 피드에는 실제 입력 폼 대신 작성 화면을 여는 한 줄짜리 버튼만 둔다. 본문이 길어져 목록과 페이지네이션을 덮던 고정 폼 문제를 구조적으로 없앤다.
- 작성 화면은 제목 없이 닉네임과 본문을 중심으로 두고, 사진·투표·AI 글쓰기를 하단 도구막대에서 어느 순서로든 조합한다.
- AI 초안은 별도 게시 흐름을 만들지 않고 같은 작성 화면의 본문 상태에 반영한다. 생성 뒤에도 본문 수정과 사진·투표 추가가 가능하다.
- `#compose`를 브라우저 히스토리에 넣어 뒤로가기로 피드에 복귀한다. 작성 화면만 닫을 때 본문·사진·투표 상태는 초기화하지 않아 실수로 닫아도 초안을 다시 열 수 있다.
- 모바일에서는 `100dvh`, safe area, 내부 스크롤을 사용하고 화면이 열려 있는 동안 바깥 문서 스크롤을 잠근다.

### 전용 게시글 수정 화면

- 게시글 수정은 목록/상세 내부의 별도 인라인 폼 대신 작성 화면과 같은 `BoardComposer`를 `edit` 모드로 재사용한다. 사진 첨부와 AI 초안도 작성 때와 같은 상호작용을 유지한다.
- 수정 draft는 기존 `boardReducer` 상태를 사용하고 저장은 기존 `PATCH /posts/{id}`를 호출한다. 게시글을 삭제 후 재생성하지 않으므로 ID, 댓글, 좋아요, 조회수는 유지된다.
- 기존 투표는 수정 화면에서 선택지 수만 안내하고 편집하지 않는다. 게시글 수정 API가 투표 엔티티를 건드리지 않으므로 이미 집계된 표도 보존된다.
- AI 생성 취소는 프롬프트 영역의 X와 화면 닫기에서 같은 `AbortController` 경로를 사용한다. 별도 지연 취소 버튼은 두지 않고 경과 시간만 표시한다.
